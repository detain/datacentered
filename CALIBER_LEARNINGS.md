# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** `start.php` loads TWO composer autoloaders: `__DIR__.'/vendor/autoload.php'` then `__DIR__.'/../../my/vendor/autoload.php'` (the shared MyAdmin vendor at `/home/my/vendor/`). When diagnosing class-loading failures, check BOTH `vendor/composer/autoload_static.php` and `autoload_classmap.php` files — a stale classmap in either one can point to paths that no longer exist.
- **[gotcha]** Workerman 5.x relocated source under a `src/` subdirectory (e.g. `workerman/workerman/src/Protocols/Http/Request.php`), while Workerman 4.x had `Protocols/Http/Request.php` directly under the package root. If `include()` warnings reference the old non-`src/` path, a classmap was generated against v4 and is now stale — run `composer dump-autoload -o` in the directory whose autoloader is stale (check both `/home/sites/datacentered/` and `/home/my/` if needed).
- **[pattern]** When the `include_path` shown in a PHP warning points at `/home/my/vendor/old_pear/...`, that means `/home/my/vendor/autoload.php` has already been loaded — so any class-resolution fault is coming from the chained MyAdmin autoloader, not just the local project's.
- **[gotcha]** The long-running `WorkerMan: master process` forks all workers at startup and keeps its Composer autoload map in memory. If `composer dump-autoload` is run on `/home/sites/datacentered/vendor/` or `/home/my/vendor/` AFTER the daemon is already running, existing workers will still resolve classes against the OLD classmap (producing `Failed to open stream` warnings for paths that no longer exist on disk). Fix: `php start.php restart` (graceful) after any composer update that touches a package used by the daemon.
- **[pattern]** To confirm a stale-autoload-in-running-process situation: compare the master PID's lstart with the mtimes of `/home/my/vendor/composer/autoload_static.php` and `/home/sites/datacentered/vendor/composer/autoload_static.php` — `ps -p <pid> -o pid,etime,lstart,cmd` vs `stat -c '%y %n' <autoload files>`. If the autoload file is newer than the process start, the process is running with a pre-update autoload map.
- **[gotcha]** A `Workerman\Protocols\Frame` error with a very large `package_length` (e.g. `1195725856` = hex `47455420` = ASCII `"GET "`) means an HTTP client connected to a port that expects Workerman's Frame protocol. This is a protocol mismatch — diagnose by converting the decimal length to hex and reading it as ASCII.
- **[gotcha]** InnoDB Group Replication certification failures (PDO exception codes 40000, 3101, 1180) mean the DB connection is still valid — do NOT reconnect `$worker_db`. Only retry the query with exponential backoff. Reserve reconnection for genuine connection-loss codes 2006 (gone-away) and 2013 (lost connection).
- **[gotcha]** In `Tasks/*.php`, the DB connection object is `$worker_db` (global, set by `start_task.php`). `Events::$db` does not exist in the Task process context — any code that reconnects `Events::$db` on cluster errors has no effect and should reconnect `$worker_db` instead.
- **[pattern]** The canonical InnoDB cluster retry wrapper for Tasks is a `$dbRetry` closure that separates cluster errors (40000/3101/1180 → exponential backoff, no reconnect) from connection errors (2006/2013 → linear backoff + reconnect `$worker_db`). All other PDO codes should release the GlobalData lock and rethrow. See `Tasks/memcached_queue_task.php` for the reference implementation.
- **[gotcha]** When debugging VPS reinstall-os password issues from the MyAdmin queue side: `GetQueue.php` builds the `$eventSubject` for queue command events — if `origrootpass` is not explicitly populated there (from `history_old_value` falling back to `vps_rootpass`), `Plugin.php:107` sees an empty `origrootpass` and generates a fresh password, silently discarding the stored one. Always verify `origrootpass` and `rootpass` are present in `$eventSubject` for password-sensitive queue actions.
- **[env]** php-cs-fixer is NOT installed under the project's `vendor/bin/` — use the global one at `/home/my/.composer/vendor/bin/php-cs-fixer`. Set `PHP_CS_FIXER_IGNORE_ENV=1` because the system PHP (8.3.x) is newer than php-cs-fixer's supported max (8.1.*). When passing explicit file paths, also pass `--config=.php-cs-fixer.dist.php` — otherwise it errors with `For multiple paths config parameter is required.` because file args override the config's own paths. Full invocation: `PHP_CS_FIXER_IGNORE_ENV=1 /home/my/.composer/vendor/bin/php-cs-fixer fix --dry-run --diff --config=.php-cs-fixer.dist.php <files>`.
- **[pattern]** `chatNotify($msg, $where = 'notifications')` in `/home/my/include/logging.php` posts to Teams (and historically Rocket.Chat) for operator notifications — default channel is `notifications`. Use it for human-visible failures (failed payment activation, permission-denied actions, etc). Always wrap calls in `if (function_exists('chatNotify'))` because the function is only present once `/home/my/include/functions.inc.php` has loaded — Tasks should call it AFTER the `require_once` of functions.inc.php and inside a try/catch so a chat outage cannot kill the task.
- **[gotcha]** Task results returned to the dispatcher arrive as a JSON STRING in the `$task_result` parameter of `dispatchTask`'s onResult callback — call `json_decode($task_result, true)` and inspect the `'return'` key to know whether the task function actually succeeded. Treating a `dispatchTask` callback firing as success-by-itself is a known footgun: payment processing previously marked rows `completed` even when `process_payment()` returned `false`. Always branch on the decoded return value when picking the next `queue_log.history_new_value` (`completed` vs `failed`).
- **[convention]** `queue_log.history_new_value` lifecycle values are `pending` → `processing` → `completed`|`failed`. Never write `completed` unconditionally on task callback fire — the task may have returned false. The `failed` state is already used (`Events.php` boardctl branch) and is the correct terminal value for unsuccessful runs.
